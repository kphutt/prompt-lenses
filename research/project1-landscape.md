# Landscape Survey: Prompt Libraries, Review Frameworks & Meta-Prompting Systems

**Date**: February 24, 2026
**Author**: Karsten Huttelmaier — co-authored with Claude
**Purpose**: Inform the design of a lens-based, model-agnostic prompt library for code and document review

## Category 1: Prompt Libraries & Collections

### Top Finds

**1. thibaultyou/prompt-library** — The best-in-class for *infrastructure*  
- **URL**: https://github.com/thibaultyou/prompt-library  
- **What it is**: A CLI-driven prompt management system with a composable fragment architecture. Prompts are stored as markdown files with YAML frontmatter, organized in `~/.prompt-library`. It supports reusable text snippets ("fragments"), variable substitution, and AI-powered metadata generation.
- **Organization**: Category folders with individual prompt files. Each prompt has structured metadata (description, tags, category) auto-generated by AI. Fragments live in `fragments/` by category and can be included in any prompt.
- **Model support**: Multi-model — supports both Anthropic and OpenAI providers, switchable via CLI config.
- **What's good**: The *fragment system* is the standout feature. It solves the composition problem — you can build complex prompts from reusable pieces (e.g., a `fragments/common/disclaimer.md` included in multiple prompts). Git-backed versioning is built-in. The prompt-as-application philosophy (treating prompts like deployable mini-apps) is architecturally mature.
- **What's lacking**: The prompts themselves are generic. The infrastructure is excellent but the content library is thin. No evaluation framework.
- **Quality**: Infrastructure is A-tier; content is B-tier.
- **Relevance to your system**: HIGH. The fragment/composition model is directly analogous to how you might compose lens components (shared preambles, output format specs, quality criteria). Study the fragment system closely.

**2. danielrosehill/System-Prompt-Library** — The most ambitious personal collection  
- **URL**: https://github.com/danielrosehill/System-Prompt-Library  
- **What it is**: 1,290+ system prompts developed by a single practitioner since summer 2024, covering agents, assistants, and workflows. Includes structured metadata (JSON schemas with booleans for agent type, workflow type, voice-first, etc.).
- **Organization**: Categorized with rich metadata. Periodic point-in-time releases. The creator treats this as a living document, rewriting rather than just versioning.
- **Model support**: Model-agnostic in principle, though many prompts are implicitly tuned for ChatGPT/GPT ecosystem.
- **What's good**: The sheer volume and the practitioner-driven evolution. Rosehill has iterated extensively and his metadata schema (with fields like "Better As Tool", "Single Turn", "External Tooling Required", "RAG Required") shows sophisticated thinking about prompt taxonomy. His philosophy — that system prompts are the "bedrock" of AI experiences — aligns with your lens approach.
- **What's lacking**: Quality is uneven across 1,290 prompts. No formal evaluation. No composition system.
- **Quality**: B+ overall; some A-tier prompts buried in a large collection.
- **Relevance**: MEDIUM. Study his taxonomy and metadata schema. The categorization thinking is more valuable than the individual prompts.

**3. abilzerian/LLM-Prompt-Library** — Experimental, multi-model  
- **URL**: https://github.com/abilzerian/LLM-Prompt-Library  
- **What it is**: A "playground" of experimental prompts using Jinja2 templates covering multiple providers (OpenAI, Anthropic, DeepSeek, Meta, Mistral, Google, xAI).
- **What's good**: Jinja2 templating is a smart choice for parameterized prompts. Genuinely multi-model.
- **What's lacking**: "Experimental" and "playground" are accurate descriptors — this is a sandbox, not a polished library.
- **Quality**: C+ to B depending on the prompt.

**4. xixu-me/prompt-library** — Best-structured template approach  
- **URL**: https://github.com/xixu-me/prompt-library  
- **What it is**: A categorized collection with an explicit template structure: Title, Description, Usage, Prompt, Example Input, Example Output, Variations, Tips. Categories include development, writing, analysis, business, creative, education, productivity.
- **Organization**: Clean folder structure with templates and quality standards documented.
- **What's good**: The template structure is exactly right — every prompt has documented expected inputs, outputs, and variations. This is what most prompt libraries lack.
- **What's lacking**: The prompts themselves are still fairly generic. Quality standards are aspirational rather than enforced.
- **Quality**: B for structure, C+ for content.

### Honorable Mentions

- **0xeb/TheBigPromptLibrary**: Educational collection of leaked system prompts from ChatGPT, Copilot, Claude, etc. Not a reusable library but invaluable for studying how the best system prompts are written. Study it for prompt craft.
- **x1xhlol/system-prompts-and-models-of-ai-tools** and **EliFuzz/awesome-system-prompts**: Collections of leaked/reverse-engineered system prompts from Cursor, Devin, Windsurf, v0, Claude Code, etc. 30,000+ lines of real production prompts. Goldmine for studying how professionals structure system prompts.
- **dontriskit/awesome-ai-system-prompts**: Curated with *analysis* — the README breaks down patterns across prompts (role definition, tool usage, safety constraints). The analysis is more valuable than the prompts.

### Key Patterns in the Best Work

1. **Prompts-as-code**: The best libraries treat prompts like software artifacts — versioned, tested, documented, composed from reusable parts.
2. **Metadata matters**: Good libraries have rich metadata (tags, categories, model compatibility, expected inputs/outputs), not just prompt text.
3. **Composition > monoliths**: Fragment/include systems (thibaultyou) and template patterns (xixu-me) dramatically improve reusability.
4. **Leaked system prompts are the real curriculum**: The production prompts from Cursor, v0, Claude Code, etc. are the most sophisticated prompt engineering artifacts in existence. They are heavily structured, use XML tags for sections, embed domain knowledge, and enforce specific workflows.

### Key Gaps

- **No library focuses specifically on review prompts** (code or document). This is a genuine gap in the landscape.
- **No library implements multi-perspective/lens approaches**. Every library organizes by *task* (code generation, summarization, etc.), not by *perspective* or *cognitive lens*.
- **No library has formal quality evaluation**. The closest is promptfoo integration, but nobody is systematically evaluating prompt quality across models.
- **Nobody is doing meta-prompt-driven prompt generation** at the library level. Individual practitioners do this; no library has industrialized it.

### Techniques to Steal

1. **thibaultyou's fragment system** — Apply this to lens composition (shared quality criteria, output format fragments, persona fragments).
2. **Rosehill's metadata taxonomy** — Adapt for lens metadata (applicable_to, perspective_type, depth_level, model_compatibility).
3. **Leaked system prompt structure** — Use XML tags for sections, explicit tool-use instructions, domain knowledge embedding, and safety constraints.
4. **xixu-me's template standard** — Every lens should have documented inputs, outputs, variations, and usage notes.

## Category 2: Review Frameworks & Lenses

### Top Finds

**1. AI Council (@mugzie/ai-council)** — Multi-agent code review with role-based perspectives  
- **URL**: https://ai-council.tech/  
- **What it is**: A commercial multi-agent code review engine with 6 specialized AI agents that review code from different perspectives, then debate disagreements.
- **Perspectives**: Senior Developer, Security Engineer, Performance Engineer, Architect, plus a general AI model (Gemini). A Judge agent synthesizes the final decision.
- **How it works**: Each agent reviews from their unique perspective → agents vote APPROVE/REVISE/REJECT with confidence scores → low-confidence agents abstain → disagreements trigger structured debates → Judge synthesizes a final recommendation.
- **What's clever**: The *confidence scoring and abstention* mechanism is brilliant — it prevents noise from perspectives that don't have relevant expertise for a given change. The structured debate on disagreements models real code review dynamics. The multi-model architecture (using different LLMs for different agents) could genuinely capture different cognitive biases.
- **Designed for**: AI use exclusively.
- **Relevance**: VERY HIGH. This is the closest existing product to your lens system applied to code review. Key differences: their perspectives are fixed and role-based; yours are configurable and cognitively diverse. Their system is runtime-only; yours is a library. Study their confidence/abstention model.

**2. Karpathy's LLM Council** — Multi-model deliberation  
- **URL**: https://github.com/karpathy/llm-council  
- **What it is**: Andrej Karpathy's implementation of a multi-model council for consensus-driven decisions. Routes the same question to GPT, Gemini, Claude, Grok, etc., and a chairman model synthesizes.
- **How it works**: All council models respond independently → chairman synthesizes a unified answer incorporating agreements and noting disagreements.
- **What's clever**: Model diversity as a feature. Different models have different training biases, so the "perspectives" emerge naturally from model differences rather than from prompt engineering.
- **Relevance**: MEDIUM. The insight — that genuine perspective diversity comes from structural differences, not just role-playing — is important. But for your system, perspective diversity through careful prompt engineering is more practical than running 4+ different models.

**3. Six Thinking Hats Applied to Technical Review**  
- **Source**: Ministry of Testing article; various AI adaptations  
- **The framework**: De Bono's six perspectives applied to software review:
  - White Hat: Facts and data (what does the code actually do?)
  - Red Hat: Gut feelings and emotions (does this feel right?)
  - Black Hat: Critical judgment (what could go wrong?)
  - Yellow Hat: Optimism (what are the benefits?)
  - Green Hat: Creativity (are there better approaches?)
  - Blue Hat: Process management (meta-review of the review itself)
- **Applied to technical review**: The Ministry of Testing article maps hats to review types — inspections use all six hats sequentially; technical reviews focus on White, Black, and Yellow; informal reviews use primarily Red.
- **PartyKit demo (blog.partykit.io)**: Matt Webb built a spatial chatroom where each room is a different "hat" with an AI facilitator, allowing teams to walk through problems room by room. Prototype-stage but conceptually interesting.
- **What's clever**: The sequencing insight — certain hat combinations work for certain purposes (Black→Green for improving designs; Yellow→Black for evaluating ideas). The Blue Hat as meta-review is directly analogous to your meta-prompting layer.
- **Relevance**: HIGH. The Six Thinking Hats is the most direct analog to your lens concept in the pre-AI world. The key insight to steal: perspectives should have defined *sequencing* for different review goals, not just be applied in parallel.

**4. ATAM (Architecture Tradeoff Analysis Method)**  
- **What it is**: A structured architecture review framework from the SEI/CMU that uses multiple *viewpoints* (stakeholder perspectives) and *quality attribute scenarios* to evaluate architecture decisions.
- **Perspectives**: Not fixed personas but stakeholder-defined quality attributes — performance, security, modifiability, availability, usability, etc. Each is evaluated against architecture decisions to identify tradeoffs.
- **What's clever**: The focus on *tradeoffs between perspectives*, not just findings within each. ATAM explicitly names the tensions (e.g., security vs. usability) and forces reviewers to address them. The concept of "sensitivity points" (architecture decisions that affect multiple quality attributes) is powerful.
- **Relevance**: HIGH. Your lens system should consider inter-lens tensions, not just independent findings. When the Security Lens and the Usability Lens disagree, that's a feature, not a bug.

### Honorable Mentions

- **prijak/Ai-council**: Open-source multi-model deliberation app with 18 pre-built persona templates across Think Tank, AI Agents, Corporate, Professional categories. More general than code review but the persona templates are interesting.
- **Agents-council (MrLesk)**: MCP-based CLI for multi-agent collaboration. Agents can peer-review each other's work. Light on structure but the inter-agent communication pattern is relevant.
- **MetaGPT's role-based approach**: Assigns distinct roles (Product Manager, Architect, Engineer) with structured communication interfaces between them. The SOPs (Standard Operating Procedures) that govern inter-role communication are worth studying.

### Key Patterns

1. **Multi-perspective works best with structured synthesis**: Every successful framework has a synthesis/judge step. Independent perspectives without synthesis create noise.
2. **Confidence scoring prevents noise**: AI Council's abstention mechanism and confidence scores are critical for keeping multi-lens output actionable.
3. **Perspective sequencing matters**: Six Hats shows that the *order* of perspectives affects quality. Creativity before evaluation produces different results than evaluation before creativity.
4. **Tradeoff identification > individual findings**: ATAM's focus on inter-perspective tensions is more valuable than any single perspective's findings.

### Key Gaps

- **Nobody has built a configurable lens library** where you can compose review perspectives for different purposes. AI Council has fixed agents; Six Hats has fixed hats.
- **No framework addresses the "same model, different perspectives" challenge**. Everyone either uses role-prompting (shallow) or different models (expensive). Nobody has cracked the problem of getting genuinely diverse perspectives from a single model through sophisticated prompt engineering.
- **No review framework includes a quality assurance layer for its own prompts**. Nobody is meta-reviewing their review lenses.

### Techniques to Steal

1. **AI Council's confidence/abstention model** — Each lens should output a confidence score and a relevance assessment. If a Security lens has nothing material to say about a CSS change, it should abstain rather than generate filler.
2. **Six Hats sequencing** — Define lens sequences for common review scenarios (e.g., "Architecture Review" = Structural→Security→Performance→Maintainability).
3. **ATAM tradeoff analysis** — After running multiple lenses, a synthesis step should explicitly identify inter-lens tensions and tradeoffs.
4. **Structured debate on disagreements** — When lenses produce contradictory recommendations, model a structured resolution process.

## Category 3: Meta-Prompting & Prompt Engineering Systems

### Top Finds

**1. DSPy (Stanford NLP)** — The heavyweight champion of programmatic prompt optimization  
- **URL**: https://dspy.ai/ / https://github.com/stanfordnlp/dspy  
- **What it is**: A framework for *programming* (not prompting) language models. You define input/output signatures and modules; DSPy automatically optimizes the prompts through bootstrapping, search, and evaluation.
- **Key insight**: Prompts should be *compiled*, not hand-written. You specify what the LM should do (signatures), and DSPy figures out how to tell it (prompts).
- **Optimizers**: BootstrapFewShot (learns from demonstrations), MIPROv2 (Bayesian optimization over instructions + few-shot examples), COPRO (hill-climbing on instructions), SIMBA (failure-aware refinement), and GEPA (reflective evolution).
- **Relevance to your system**: DSPy's philosophy directly challenges hand-crafted prompt libraries. If you can specify "review this code for security issues and produce structured findings" as a DSPy signature, it can potentially optimize the prompt better than you can write it. However, DSPy is designed for *production pipelines*, not for *human-readable prompt libraries*. The paradigms are complementary: you craft the lens philosophy and structure; DSPy could optimize the execution prompts.
- **Practical applicability**: MEDIUM-HIGH. DSPy requires training data and evaluation metrics, which you'd need to build. But even 30-50 examples per lens could yield meaningful optimization. The 20/80 train/val split recommendation (reversed from ML convention) is particularly relevant for prompt optimization.

**2. GEPA (Reflective Text Evolution)** — DSPy's cutting-edge optimizer  
- **URL**: https://github.com/gepa-ai/gepa  
- **What it is**: An optimizer that uses LLM reflection to analyze what worked/didn't in program execution, then proposes improved prompts. Endorsed by Shopify CEO Tobi Lutke as "severely under-hyped."
- **Key insight**: "Actionable Side Information" (ASI) — diagnostic feedback from evaluators that serves as a text-optimization analog of a gradient. Instead of numeric loss, you get natural language explanations of what went wrong.
- **Results**: GPT-4.1 Mini from 46.6% → 56.6% on AIME 2025. ChainOfThought from 67% → 93% on MATH benchmark.
- **Relevance**: HIGH. GEPA could be used to optimize your lens prompts against evaluation datasets. The "reflective" approach (analyzing failures to generate improvements) maps perfectly to iterating on lens quality.

**3. Promptfoo** — The testing framework your system needs  
- **URL**: https://www.promptfoo.dev/ / https://github.com/promptfoo/promptfoo  
- **What it is**: Open-source CLI for testing, evaluating, and red-teaming LLM prompts. Declarative YAML configs, model-agnostic, CI/CD integration, side-by-side comparison of prompts across models.
- **Key capabilities**: Batch testing across multiple models and prompts. Assertions (contains, matches, LLM-rubric, G-Eval). Cost and latency measurement. Red teaming (prompt injection, PII leak detection). Human-readable output with web dashboard.
- **Relevance**: VERY HIGH. This is the evaluation layer your meta-prompting system needs. You can define test cases for each lens (given this code input, the Security lens should identify X, Y, Z) and run them across models. This gives you data-driven quality assurance rather than vibes-based assessment.
- **Integration opportunity**: Define promptfoo configs for each lens. Use G-Eval criteria to evaluate lens output quality. Run regression tests when you update lens prompts.

**4. Salesforce's Promptomatix** — Automated prompt optimization  
- **URL**: https://github.com/SalesforceAIResearch/promptomatix  
- **What it is**: A framework that automates prompt optimization using DSPy. Takes a task description, determines the task type, selects appropriate DSPy modules and metrics, and iteratively refines prompts based on feedback.
- **Key insight**: Task-specific optimization — automatically selecting the right optimization strategy based on what you're trying to do. The paper (arXiv:2507.14241) includes detailed prompt engineering guidelines in Appendix B.
- **Relevance**: MEDIUM. More relevant if you want to automate lens prompt optimization rather than hand-craft each one.

**5. Stanford/OpenAI Meta-Prompting Research (Jan 2024)**  
- **What it is**: Joint research proposing a meta-prompting approach where a "conductor" LLM orchestrates multiple "expert" instances, each with specialized prompts. Reports 17.1% improvement over standard prompting.
- **Key insight**: The structure-first approach — meta-prompting focuses on *syntactical and structural* dimensions of reasoning rather than content-driven few-shot examples. This means you can be more token-efficient while achieving better results.
- **Relevance**: HIGH. This directly validates your multi-layer approach (meta-prompts that generate lens prompts). The research supports the idea that structural guidance outperforms example-driven approaches.

### Honorable Mentions

- **TextGrad**: Uses natural-language feedback as "textual gradients" for iterative prompt improvement. Published in Nature (2025). More nuanced than numeric optimization but slower and more expensive.
- **OpenAI's Meta-Prompting Cookbook**: Practical tutorial on using o1/o-series models to improve prompts for smaller models. Simple and effective for one-off prompt improvement.
- **Anthropic's Prompt Engineering Best Practices** (claude.com/blog): Covers the shift from prompt engineering to context engineering. Key insight: "the best prompt isn't the longest or most complex — it's the one that achieves your goals reliably with the minimum necessary structure."
- **Prompting Guide (promptingguide.ai)**: DAIR.AI's comprehensive reference covering meta-prompting, CoT, ReAct, and more. Good as a reference but not deep on any single technique.

### Key Patterns

1. **The compile-don't-write paradigm**: DSPy's core thesis — that prompts should be programmatically optimized rather than hand-written — is gaining rapid adoption. Even if you hand-craft your lens philosophy, the execution prompts can be optimized.
2. **Evaluation-driven development**: Every serious prompt optimization approach requires metrics and test data. Vibes-based prompt writing doesn't scale.
3. **Meta-prompting works**: The research consistently shows 10-20%+ improvement from using LLMs to generate/refine prompts. Your meta-prompt layer is well-justified.
4. **Reflection > brute-force search**: GEPA's reflective approach (analyzing failures) outperforms random search or hill-climbing for prompt optimization.

### Key Gaps

- **No meta-prompting system is designed for *libraries* of prompts**. Everything is optimized for single-task pipelines.
- **No evaluation framework exists for *review quality***. Promptfoo can test whether output contains certain strings, but evaluating whether a code review is *insightful* requires domain-specific rubrics that nobody has built.
- **The quality assurance layer you're building (meta-prompts that evaluate lens quality) doesn't exist anywhere else.**

### Techniques to Steal

1. **Promptfoo as your evaluation backbone** — Define YAML test configs for each lens with model-graded evaluation criteria.
2. **GEPA's reflective optimization** — After evaluating lens output, use the failures to generate improved lens prompts.
3. **DSPy's signature concept** — Define each lens as a signature (inputs → outputs) rather than a monolithic prompt. This makes lenses more composable and optimizable.
4. **Structured output parsing** — Every lens should produce structured output that can be programmatically evaluated, not free-text prose.

## Category 4: AI-Assisted Code Review Tools

### Top Finds

**1. CodeRabbit** — The market leader, highly configurable  
- **URL**: https://www.coderabbit.ai/  
- **What it is**: AI code review platform with 2M+ repos connected, 13M+ PRs reviewed. Integrates with GitHub, GitLab, Bitbucket, Azure DevOps. Has PR, CLI, and IDE modes.
- **How it structures analysis**: Combines 40+ linters/SAST tools with LLM analysis. Uses a verification agent to cut linter noise. Reviews are context-aware (understands the full codebase, not just the diff).
- **Custom perspectives**: Yes — path-based instructions (`**/*.js` → "Review against Google JS style guide"), code guidelines (auto-detects CLAUDE.md, .cursorrules, etc.), AST-grep rules for structural patterns, and learnable feedback (improves from accepted/rejected suggestions).
- **"Lens" equivalent**: Path instructions + code guidelines + configurable "nitpickiness" level. Not truly multi-perspective but highly customizable.
- **What they've learned**: Noise is the #1 problem. Too many comments cause review fatigue. Their verification agent (an LLM that evaluates linter output before presenting it) is specifically designed to reduce noise. Configuration inheritance across repos is key for scaling.
- **Relevance**: HIGH. CodeRabbit represents the state of the art in *productized* AI code review. Key lesson: review quality is measured by signal-to-noise ratio, not by volume of findings. Your lenses should be designed to minimize noise, not maximize coverage.

**2. Qodo (formerly CodiumAI)** — Rule-based perspectives  
- **URL**: https://www.qodo.ai/  
- **What it is**: AI code integrity platform focusing on review, testing, and writing. Differentiator is configurable rulesets.
- **How it structures analysis**: Encode rules for style, patterns, frameworks, security, compliance. Review agents check each diff against these rules. Learns from PR history, comments, and discussions.
- **"Lens" equivalent**: Rulesets function as review perspectives. You can define a "security ruleset" and a "style ruleset" and apply them independently. Closer to your lens concept than most tools.
- **Relevance**: MEDIUM-HIGH. The ruleset-as-perspective model is instructive. Key insight: rules need to be *centrally managed* and *applied across repos* to maintain consistency.

**3. Greptile** — Codebase-aware review  
- **URL**: https://www.greptile.com/  
- **What it is**: AI code review that indexes your entire codebase for context-aware feedback. Claims to understand "your entire codebase."
- **How it structures analysis**: Single-pass LLM analysis with full codebase context via retrieval. No explicit multi-perspective.
- **Relevance**: MEDIUM. The codebase-awareness insight is important — review quality depends heavily on understanding the broader context, not just the diff. Your lenses might benefit from codebase context injection.

**4. Cursor BugBot** — Focused on critical bugs  
- **What it is**: Cursor's code review tool that specifically targets high-severity bugs rather than style issues. The opposite of CodeRabbit's comprehensive approach.
- **Key insight**: Laser focus on critical defects outperforms comprehensive analysis for high-stakes code. Different review goals need different approaches.
- **Relevance**: MEDIUM. Validates your lens approach — different lenses for different purposes, not one all-encompassing review.

### Key Patterns

1. **Noise is the enemy**: Every mature code review tool has had to solve the noise problem. More findings ≠ better review.
2. **Customizable rules > fixed analysis**: Teams need to encode their own standards. Generic review doesn't scale.
3. **Context is king**: Codebase-aware review (Greptile, CodeRabbit) dramatically outperforms diff-only analysis.
4. **Learning from feedback**: The best tools improve from accepted/rejected suggestions. Your lens system could benefit from a similar feedback loop.

### Key Gaps

- **No tool implements true multi-perspective review** with independent lenses that can disagree. CodeRabbit comes closest with path instructions, but it's one unified analysis, not genuinely separate perspectives.
- **No tool separates the review concerns architecturally** — security findings, performance findings, and style findings all come from the same analysis pass.
- **No tool allows users to bring their own review prompts/lenses**. Custom rules, yes; custom cognitive perspectives, no.

### Techniques to Steal

1. **CodeRabbit's verification agent** — A noise-reduction layer that evaluates each finding before presenting it. Apply this to each lens: is this finding material enough to report?
2. **Confidence scoring from AI Council** — Every finding should carry a confidence score.
3. **Qodo's centrally-managed rulesets** — Your lens library should support organization-level customization.
4. **Cursor BugBot's focus philosophy** — Design lenses with clear scope boundaries. A Security lens should not comment on style.

## Category 5: Writing & Editing Frameworks

### Top Finds

**1. The Professional Editing Stack** — A natural multi-pass framework  
The professional editing world has independently evolved the exact multi-lens pattern your system implements. The established editing levels are:

- **Developmental Editing** (Big Picture Lens): Story/argument structure, logical flow, completeness, audience appropriateness, organization. Does the document achieve its purpose?
- **Structural Editing**: Chapter/section organization, pacing, transitions, information hierarchy. Is the document well-organized?
- **Line Editing** (Style Lens): Sentence-level craft — word choice, rhythm, tone, voice, clarity. Is each sentence effective?
- **Copy Editing** (Correctness Lens): Grammar, punctuation, spelling, consistency, adherence to style guide (Chicago, AP, etc.). Is the writing mechanically correct?
- **Proofreading** (Final Check Lens): Typos, formatting, page numbers, references. Is the document clean?

**Key insight**: These levels are applied *in sequence*, from macro to micro. You don't copy-edit before developmental editing because the structure may change. This sequencing principle directly applies to code review (don't nitpick style before addressing architecture).

**What's clever**: Each editing level has a different *cognitive mode*. Developmental editing requires stepping back and seeing the whole; line editing requires sentence-level immersion; copy editing requires mechanical rule-following. These are genuinely different thinking patterns, not just different checklists.

**Relevance**: VERY HIGH. This is the most mature existing example of a multi-lens review system. The sequencing, the distinct cognitive modes, and the principle that each level has its own scope boundaries are all directly applicable.

**2. Federal Plain Language Guidelines** — A quality standard that functions as a review lens  
- **URL**: https://plainlanguage.gov/guidelines/  
- **What it is**: Legally mandated writing standards for US federal documents. Covers audience analysis, logical organization, plain language techniques (active voice, short sentences, common words), and usability testing.
- **Key principles**: Write for your audience; organize for your reader, not yourself; use plain language; test with real readers.
- **As a review lens**: You could implement the entire Plain Language Guidelines as a single document review lens. Each guideline maps to a specific, evaluable criterion.
- **Relevance**: HIGH. This is the most concrete, specific example of a "writing quality lens" — it's essentially a checklist with explanations that could be directly encoded as a prompt.

**3. Document Review Process (Technical Documentation)**  
The technical documentation world uses a structured multi-reviewer process that maps to your lens concept:

- **Self-Review**: Writer's first pass — structure, clarity, typos.
- **Peer Review**: Relevance, content structure, intended flow.
- **SME (Subject Matter Expert) Review**: Technical accuracy, completeness, correctness.
- **Editorial Review**: Quality standards, style format, language polish.
- **Compliance Review**: Regulatory adherence, legal requirements, industry standards.
- **Final Review**: Formatting consistency, overall quality, publication readiness.

**Key insight**: Each reviewer type has a specific *question they're trying to answer*: the SME asks "is this correct?", the editor asks "is this well-written?", the compliance reviewer asks "is this legal?" This maps directly to your lens model.

**4. The Economist Style Guide / Chicago Manual of Style**  
- These aren't review frameworks per se, but they function as the *knowledge base* for specific lenses. A "house style" lens would be parameterized by the style guide it follows.
- **Relevance**: MEDIUM. The insight is that writing lenses need an authoritative reference — you can't just say "check style." You need to specify *whose* style.

### Honorable Mentions

- **SEC Plain English Handbook**: Financial document-specific plain language guide. Could be a domain-specific writing lens.
- **NARA Style Guide**: Federal records-specific writing standards.
- **WordRake**: Commercial tool that automates Federal Plain Language compliance. Implements 90% of PlainLanguage.gov's simplification suggestions programmatically.

### Key Patterns

1. **Sequential, macro-to-micro review**: Always review structure before details. This is universal across editing, architecture review, and code review.
2. **Each pass has a single cognitive focus**: Don't mix concerns. A style pass shouldn't flag structural issues.
3. **Authority-backed criteria**: The best writing review is grounded in a specific style guide or standard, not vague "write better."
4. **Testing with real users**: The ultimate measure of document quality is whether the audience can use it. Plain Language Guidelines mandate usability testing.

### Key Gaps

- **Nobody has encoded professional editing levels as AI prompts**. This is a massive gap. A well-crafted "Developmental Editing Lens" prompt would be immediately useful to millions of writers.
- **No AI writing review tool uses the multi-pass editing model**. They all do a single omnibus analysis.
- **No bridge between code review and document review frameworks**. The same multi-lens principles apply to both, but nobody has unified them.

### Techniques to Steal

1. **The editing stack sequencing** — Apply macro-to-micro ordering to both code and document lenses.
2. **Single cognitive focus per lens** — Each lens should have one clear question it's answering.
3. **Authority-backed criteria** — Parameterize lenses with specific standards (OWASP for security, Plain Language Guidelines for clarity, Google Style Guide for code style).
4. **The SME/Editor/Compliance division** — Map these roles to code review perspectives (Domain Expert, Code Craft, Compliance/Regulatory).

## Synthesis

### The 10 Most Important Things to Look At (Ranked)

1. **thibaultyou/prompt-library** — Study the fragment composition system. This is the best infrastructure model for a reusable prompt library. Adopt the fragment pattern for your lenses.

2. **AI Council (@mugzie/ai-council)** — The closest existing implementation of your vision for code review. Study the confidence scoring, abstention, and structured debate mechanisms.
